# T5.4 — README showpiece + OSS-readiness bundle (design)

**Date:** 2026-07-21
**Task:** T5.4 (OSS readiness) — closes the doc half of v0.1.
**Predecessor spec:** [2026-07-21-evidence-campaign-and-authored-docs-design.md](2026-07-21-evidence-campaign-and-authored-docs-design.md) §3.3.

## 1. Goal

Turn the README from a competent-but-generic technical doc into an authored
**showpiece** — one that looks distinctive, explains the author's reasoning in
his own voice, and shows the real artifact — **without betraying the honest,
un-hyped voice that is the product's identity.** suMCP catches agents overselling
their own work; its README therefore cannot open with a marketing scream. The
design goal is: bold *aesthetic*, evidence-bound *claims*.

Then finish the rest of the OSS-readiness bundle (LICENSE, CONTRIBUTING,
CHANGELOG, SECURITY.md, docs/metrics.md, rustdoc, issue templates) at
checklist depth.

## 3. README design (deep)

### 3.1 Structure

One scannable page:

```
[ HERO TITLE WORDMARK — authored SVG (V6) ]
tagline: "The agent tells you what it built.
          suMCP tells you what it actually did."
[ badge row ]  (§3.5)
┌─ HERO SCREENSHOT — the Win9x report (§3.4) ─┐
honest status line: "v0.1 pre-release — validated on the author's own
                     projects; ranking accuracy not yet systematically measured"

## Why I built this      ← first person, author's voice (§3.2)
## How it works          ← hand-drawn PIPELINE diagram (hero) + existing prose
## The numbers           ← hand-drawn TOKEN diagram + the ~800× line
## Install               ← existing dry-run → apply one-liner, unchanged
## Quickstart            ← existing
## The six tools         ← existing table, unchanged
## Limitations           ← kept prominent (honesty tool earns trust here)
## License               ← dual MIT OR Apache-2.0 (§4)
```

Existing sections that already work (Install, Quickstart, the six-tool table,
Limitations) are preserved as-is; T5.4 adds the title, tagline, badges,
screenshot, "Why I built this", and the two diagrams, and reconciles the
License line to the dual license.

### 3.2 "Why I built this" (methodology)

A few tight first-person paragraphs, drafted from the already-locked thesis and
edited by the author until it reads as his, not the assistant's. Beats to cover:

- **Comprehension debt** — shipping agent-written code you don't fully understand.
- **Self-report bias** — when you ask an agent "what did we struggle with?", it
  answers from a lossy, self-flattering memory, or by re-reading the whole
  transcript.
- **Evidence over narrative** — the transcript is behavioral evidence: every
  edit, failure, and pushback, ordered and timestamped. It survives compaction
  and self-report bias.
- **Why deterministic / no-LLM / read-only** — the parse is near-free and
  reproducible; the connected agent is the only intelligence layer.

Claims here stay evidence-bound (V1): no unbacked superiority claims; hedges kept
where the predecessor audits require them.

### 3.3 Hand-drawn diagrams (V3)

**Delivery:** authored **sketch-style SVGs** (wobbly strokes, hand-lettered
look — the excalidraw/rough aesthetic), committed under `docs/assets/`, embedded
as images so they render reliably on GitHub and match the retro report.
Chosen over Mermaid's hand-drawn mode because GitHub's native Mermaid renderer
does not honor the `handDrawn` look — an embedded SVG guarantees the look.

**Honesty-to-code (D7):** for the pipeline, also keep a plain Mermaid source
block in `docs/` (or fenced in the README) so the architecture stays diffable;
the *shown* image is the hand-drawn SVG.

1. **How it works (hero):**
   `session.jsonl → deterministic Rust parser → session graph → 6 MCP tools →
   your agent, cited`, with `no LLM · no network · read-only` hand-lettered
   beneath. Mirrors the real pipeline `locate → ingest → model → signals →
   score → Report`.
2. **The token headline:** a giant transcript block vs the tiny ~150–290-token
   payload, drawn to scale, labeled `median ~800× smaller`. Only numbers the
   provenance audit backs; the footnote defining the measurement
   (`session_overview` payload vs full transcript, `chars/3.5`) stays.

### 3.4 Hero screenshot (V4)

The report as rendered by `sumcp --html`, regenerated from a **clean synthetic
session** — a small hand-built fixture with realistic-but-fake paths (no
`Tene_Volume` / `3DAeroRelief` / real project internals) and a clear struggle
story (churn + rework + a blind-write attempt + re-read thrash, so every panel
has content). Build the fixture, render `--html`, screenshot at README scale
(~1100px wide), commit the PNG under `docs/assets/`.

The synthetic fixture doubles as a stable regression input; it must not leak any
real path and should exercise each report panel (Overview, Timeline, Struggle
areas, Blind spots, at least one File story with evidence).

### 3.5 Badges (V1-honest)

`license: MIT OR Apache-2.0` · `Rust` · `deterministic · no LLM · no network`
· `v0.1 pre-release`. **No crates.io badge** until actually published. Add a CI
badge only if a public CI workflow exists and is green.

### 3.6 Hero title wordmark (V6)

Authored SVG wordmark for "suMCP", bold retro style bridging the navy/teal
Win9x titlebar and the hand-drawn diagrams. Produce 2–3 options at build time
(e.g. a Win9x-titlebar riff, a hand-lettered version, a chunky pixel version);
author selects one. Committed under `docs/assets/`; the others discarded.

## 4. Mechanical OSS docs (checklist depth)

- **LICENSE** — dual **MIT OR Apache-2.0** (locked). Add both license files
  (`LICENSE-MIT`, `LICENSE-APACHE`) per Rust convention; reconcile the README
  License section and `Cargo.toml` `license` field to `MIT OR Apache-2.0`.
- **CONTRIBUTING.md** — dev setup (`cargo build`, `cargo test --workspace`),
  the **fixture-donation flow via the sanitizer**, and how to add a signal
  (where detectors live, the tier/exact-vs-heuristic/confidence contract).
- **CHANGELOG.md** — Keep-a-Changelog format; `0.1.0` under Unreleased.
- **SECURITY.md** — the read-only / no-network / writes-only-under-`$HOME`
  posture; how to report an issue.
- **docs/metrics.md** — every *shipped* metric: definition, tier,
  exact-vs-heuristic, known limits; distilled from `docs/metrics-spec.md` and
  honest about what is heuristic. Must match what the code actually fires.
- **rustdoc pass** — `#![warn(missing_docs)]` on `sumcp-core`, doc every public
  item, `cargo doc` builds clean; doc-tests run in `cargo test`.
- **GitHub issue templates** — bug + feature, plus a **"signal felt wrong"**
  template that asks for a sanitized fixture and the surprising ranking.

## 5. Acceptance

- A stranger goes from README → a working debrief without reading source.
- README renders on GitHub with: wordmark, tagline, badges, hero screenshot,
  both hand-drawn diagrams, and the "Why I built this" section — no broken image
  links, no real project paths in the screenshot.
- Every quantitative claim in the README traces to a measured number or an
  explicit hedge (V1); numbers match the provenance audit.
- `cargo doc` clean; `#![warn(missing_docs)]` on `sumcp-core` with no warnings;
  doc-tests pass in `cargo test --workspace`.
- `Cargo.toml`, `LICENSE*`, and the README License section all agree on
  `MIT OR Apache-2.0`.

## 6. Out of scope

- Report redesign (V2 keeps the Win9x look).
- Hosting a live example (V4 is a screenshot only).
- crates.io / GitHub publishing — an ask-first boundary (SPEC §7), not this task.
- Cross-session features (v0.2).

## 7. Open items / risks

- **Synthetic fixture realism:** the struggle story must look plausible, not a
  strawman, or the hero screenshot reads as fake. Hand-build it with care.
- **Wordmark taste:** subjective; mitigated by producing options and letting the
  author choose (V6).
- **`docs/metrics.md` drift:** it must describe metrics as the code actually
  fires them (incl. the empirical amendments — e.g. blind-write reframed as
  attempts, `flip`/`true_revert` rare), not as the original metrics-spec wished.
- **Number provenance:** any ρ/% or ratio that did not survive the provenance
  audit must be cut or softened; that is an acceptable outcome, not a failure.
